# План: QA

**Роль:** автоматизация тестирования, тестовые данные, контроль качества, документация.
**Стек:** pytest + coverage, Playwright, Locust (нагрузка), Markdown/PDF (User Guide).
**Директория:** `backend/tests/`, `tests/fixtures/`, e2e-тесты.

Стадии по фичам-поставкам. Гейт = ① runnable-проверка ② чеклист Done (ID ТЗ) ③ ревью sub-agent'ом.

Ревью гейта:
> Task(subagent_type="review-code", prompt="Проверь стадию N плана qa: покрывают ли тесты <ID>, есть ли негативные кейсы и edge-cases, не флейки ли. Верни confirmed-findings.")

---

## Стадия 1 — Тестовые данные (QA-03)

**Задачи**
- Набор в `tests/fixtures/`: корректные PDF/DOCX, пустые файлы, битое форматирование, нестандартные шрифты.
- Файл-манифест: что каждый образец проверяет.

**Гейт 1**
- Команда: `ls tests/fixtures/` → присутствуют все 4 категории.
- Чеклист: [ ] QA-03 корректные [ ] пустые [ ] битые [ ] нестандартные шрифты [ ] манифест описывает назначение.
- Ревью: sub-agent review-completeness — покрыты ли все ветки парсера (PDF/DOCX × ok/пустой/битый).

---

## Стадия 2 — Юнит-тесты + покрытие ≥50% (QA-01)

**Задачи**
- pytest на парсинг (PDF/DOCX), чанкинг, валидацию (формат/размер).
- Негативные кейсы: >20 МБ, неверный формат, пустой файл.
- Замер покрытия.

**Гейт 2**
- Команда: `pytest backend/ --cov=app --cov-report=term-missing` → **coverage ≥ 50%**, все зелёные.
- Чеклист: [ ] QA-01 юнит-тесты есть [ ] покрытие ≥50% [ ] негативные кейсы валидации [ ] чанкинг проверен по границам.
- Ревью: sub-agent review-code — тесты проверяют поведение, а не реализацию; нет ложно-зелёных.

---

## Стадия 3 — E2E Playwright (QA-02)

**Задачи**
- Сквозной сценарий: загрузка документа → индексация → поиск → отображение результатов.
- Проверка подсветки и наличия полей в карточке.

**Гейт 3**
- Команда: `npx playwright test` против поднятого `docker compose` → сценарий зелёный.
- Чеклист: [ ] QA-02 критический путь автоматизирован [ ] ждёт статус «Готово» перед поиском [ ] проверяет карточку+подсветку [ ] не флейки (3 прогона подряд).
- Ревью: sub-agent review-code — устойчивые селекторы/ожидания вместо sleep; изоляция тестовых данных.

---

## Стадия 4 — Нагрузочные тесты (QA-04)

**Задачи**
- Locust: 50 одновременных пользователей, поисковые запросы.
- Отчёт: время отклика (p50/p95), RPS, ошибки.

**Гейт 4**
- Команда: `locust -f loadtest.py --headless -u 50 -r 10 -t 1m` → отчёт сформирован.
- Чеклист: [ ] QA-04 50 юзеров [ ] отчёт с временем отклика [ ] зафиксированы p95 и % ошибок.
- Ревью: sub-agent review-performance — репрезентативность сценария; влияние Redis-кеша на цифры отмечено.

---

## Стадия 5 — Оценка качества поиска Precision@3 (QA-05)

**Задачи**
- 10 эталонных запросов с известным ожидаемым документом.
- Проверить, что нужный документ в топ-3; оформить таблицей (запрос / ожидаемый / позиция / hit).

**Гейт 5**
- Команда: скрипт прогоняет 10 запросов к `/search`, печатает таблицу + итоговый Precision@3.
- Чеклист: [ ] QA-05 10 запросов [ ] таблица результатов [ ] метрика посчитана [ ] расхождения прокомментированы.
- Ревью: sub-agent review-completeness — корректность методики (эталон задан заранее, не подогнан).

---

## Стадия 6 — Руководство пользователя (QA-06, критерий «Документация»)

**Задачи**
- `docs/user-guide.md` (или PDF): загрузка, поиск, подсветка, скриншоты.

**Гейт 6 (финальный)**
- Команда: пройти по инструкции на чистом стенде — все шаги воспроизводятся.
- Чеклист: [ ] QA-06 гайд покрывает все пользовательские сценарии [ ] скриншоты актуальны [ ] соответствует реальному UI.
- Ревью: sub-agent review-completeness — гайд покрывает все FE-требования, нет расхождений с системой.
