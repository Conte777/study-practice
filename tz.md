# Разработка интеллектуальной поисковой системы по внутренней базе знаний университета

## 1. ОБЩИЕ ПОЛОЖЕНИЯ

### Цель практики

Закрепление и углубление теоретических знаний, полученных при изучении дисциплин через разработку полнофункционального программного продукта с использованием современных технологий.

### Задачи практики

1. Разработать архитектуру приложения на основе микросервисного подхода.
2. Реализовать систему полнотекстового поиска с использованием Elasticsearch.
3. Создать пользовательский интерфейс для загрузки документов и поиска по ним.
4. Настроить среду исполнения с использованием контейнеризации.
5. Автоматизировать процесс тестирования и развертывания.
6. Обеспечить документирование разработанного программного обеспечения.

### Состав команды и распределение ролей

Каждая команда состоит из 4 студентов. Распределение ролей осуществляется на первом организационном собрании.

| № | Роль | Код роли | Функциональные обязанности |
|---|------|----------|------------------------------|
| 1. | Бэкенд-инженер | BE | Разработка API, интеграция с Elasticsearch, бизнес-логика, работа с базами данных |
| 2. | Фронтенд-инженер | FE | Разработка пользовательского интерфейса, верстка, клиент-серверное взаимодействие |
| 3. | DevOps-инженер | DO | Контейнеризация, CI/CD настройка, мониторинг, управление окружением |
| 4. | QA-инженер | QA | Автоматизация тестирования, подготовка тестовых данных, контроль качества, документация |

## 2. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ К СИСТЕМЕ

### Общие требования

Система должна представлять собой веб-приложение, обеспечивающее:

- Авторизацию пользователей по упрощенной схеме (логин/пароль, без сложных ролей, либо вовсе без авторизации для прототипа).
- Загрузку документов в форматах PDF и DOCX.
- Полнотекстовый поиск по загруженным документам с ранжированием результатов.
- Отображение найденных фрагментов с подсветкой совпадений.
- Сохранение истории поисковых запросов.

### Требования к бэкенд-части

#### Модуль загрузки и обработки документов

| ID требования | Описание | Приоритет |
|---|---|---|
| BE-01 | Реализовать REST API эндпоинт `POST /api/v1/documents/upload` для загрузки файлов | Высокий |
| BE-02 | Проводить валидацию файлов: формат (PDF, DOCX), размер (не более 20 МБ). При несоответствии возвращать HTTP 400 с описанием ошибки | Высокий |
| BE-03 | Генерировать уникальный идентификатор (UUID) для каждого загруженного документа | Высокий |
| BE-04 | Реализовать извлечение текста из загруженных документов с использованием библиотек: pdfplumber (для PDF), python-docx (для DOCX) | Высокий |
| BE-05 | Выполнять разбиение извлеченного текста на чанки (сегменты) по 1000 символов с перекрытием 100 символов между соседними чанками | Средний |

#### Модуль индексации и поиска

| ID требования | Описание | Приоритет |
|---|---|---|
| BE-06 | Настроить Elasticsearch с русскоязычным анализатором (analysis-ru). Создать индекс documents с маппингом для полнотекстового поиска | Высокий |
| BE-07 | Реализовать функцию индексации: каждый чанк документа должен быть сохранен в Elasticsearch с метаданными (file_name, page_number, chunk_id, text) | Высокий |
| BE-08 | Реализовать эндпоинт `GET /api/v1/search?q={query}`. Запрос должен выполняться к Elasticsearch с использованием multi_match по полю text | Высокий |
| BE-09 | Результаты поиска должны возвращаться в формате JSON с полями: chunk_id, file_name, page, text, score (релевантность) | Высокий |
| BE-10 | Реализовать кеширование частых запросов через Redis (TTL = 5 минут). Если один и тот же запрос поступает повторно, ответ должен браться из кеша | Низкий |

### Требования к API

- Все эндпоинты должны быть задокументированы через OpenAPI 3.0 (Swagger UI доступен по /docs).
- Ответы должны содержать корректные HTTP-статусы:
  - 200 OK — успешное выполнение.
  - 400 Bad Request — ошибка валидации.
  - 404 Not Found — ресурс не найден.
  - 500 Internal Server Error — внутренняя ошибка сервера.

### Требования к фронтенд-части

|---|---|---|

| FE-02 | Отображать прогресс загрузки и индексации каждого файла в виде прогресс-бара. Состояния: Загрузка..., Индексация..., Готово, Ошибка | Высокий |
| FE-03 | Отображать список уже загруженных документов с их названиями, датой загрузки и статусом | Средний |

#### Интерфейс поиска

| ID требования | Описание | Приоритет |
|---|---|---|

| FE-07 | Реализовать пагинацию (по 10 результатов на страницу) либо бесконечный скролл (загрузка новых результатов при скролле вниз) | Средний |
| FE-08 | При отсутствии результатов выводить сообщение: «По вашему запросу ничего не найдено. Попробуйте изменить формулировку» | Низкий |

## 3. ТЕХНИЧЕСКИЕ ОГРАНИЧЕНИЯ

### Технологический стек (необязательный к применению, но желательный)

Python, FastAPI, PostgreSQL, Elasticsearch, Redis, TypeScript, React (или Vue 3), Docker / Docker Compose, GitHub Actions, Pytest, OpenAPI (Swagger)

### Требования к качеству кода

1. Код должен быть оформлен в соответствии с руководствами по написанию кода (PEP 8 и т.п.)
2. Все публичные функции и классы должны содержать описание, параметры, возвращаемые значения.
3. Названия переменных и функций должны быть понятными и отражать суть выполняемых действий.
4. В репозитории запрещено хранение бинарных файлов (за исключением папки /tests/fixtures для тестовых документов).
5. Каждый коммит должен иметь осмысленное сообщение: `feat: add document upload endpoint`, `fix: correct PDF parsing encoding`.

Структура репозитория должна соответствовать следующему шаблону:
